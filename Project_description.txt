DESCRIPTION
This project nowcast (same-day) and forecast (next-day) cryptocurrency volatility based on Bitcoin price data, the Fear & Greed Index, and Google Trends data. The project is structured in different pipelines:
	- Data pipeline: data fetching, pre-processing, merging and cleaning.
	- EDA & Feature engineering: exploring data distribution and summary statistics; builds new features from existing one, defines cluster through k-means and assess dimensionality reduction thanks to PCA. 
	- Model tuning & selection: light-tuning of several different models (regression, tree-based, LSTM and transformers) and top-performing model assessment and selection.
	- Model Deployment: by inputing the date, the model forecast volatility for today and tomorrow.
	- Periodic Model re-training: The model(s) automatically re-train monthly based on an expanding window strategy. 
-> TARGET VARIABLE: Exponentially Weighted Moving Average (EWMA) Volatility.
-> MODEL DEPLOYMENT: model deployed on Hugging Face spaces.

TARGET VARIABLE EXPLANATION - EWMA vs n-days rolling std volatility
When forecasting financial volatility, the literature typically considers two common proxies:
	- N-day Rolling Standard Deviation: Computes the standard deviation of returns over a fixed window of n days, assigning equal weight to all observations in the window.
	- Exponentially Weighted Moving Average (EWMA) Volatility: Calculates volatility using exponentially decaying weights, giving more importance to recent returns while still accounting for older data.
In this case study, the goal is to predict Bitcoin’s future volatility. Given that crypto markets are known for their extreme and fast-moving volatility, two key considerations arise:
	- Recent market behavior tends to be more informative than older, more distant observations.
	- Excessive noise in the volatility series can harm model performance, especially for machine learning models that benefit from smoother target signals.
These observations suggest that EWMA volatility is better suited as the target variable. It excels at:
	- Smoothing the volatility signal (reducing noise).
	- Prioritizing recent trends through decaying weights.
Therefore, the 5-day EWMA volatility is selected as the target. A span=5 gives approximately 33% weight to the most recent return, ensuring responsiveness to market shifts while still retaining some memory of past movements.
Note. For completeness, the same forecasting analysis was also conducted using 5-day, 21-day, and 30-day rolling standard deviation volatility as targets. All resulted in significantly weaker model performance, further validating the choice of EWMA. 


MODELING TUNING & SELECTION PROCEDURE
Through this analysis, several different models are implemented in order to choose a final top-performing model. In details, the procedure will go as follows:  
	- Choose a set of model to implement. This choice is flexible, and additional models can be introduced at any stage.  
	- Perform Light tuning and fit (with TimeSeriesSplit validation to keep temporal-ordering during validation) each model on the training dataset (X_scal_tr and y_scal_tr). Light-tuning uses a narrow hyperparameter grid to save computational resources and time.
	- Evaluate each model on the validation set (X-val_scal and y_val_scal) using consistent scoring metrics (named scoring).
	- Select the top 2-3 performing models based on evaluation metrics. 
	- Fine-tune the top performing models on the training set. This is done with a broader hyper-parameters grid.
	- Re-evaluate the top models on the validation set. 
	- Choose the overall top-performing model. This is the “champion” model.
	- Evaluate this model on the unseen dataset, i.e the testing set (X_scal_tt, y_scal_tt), using the same metrics as before.
	- If performance is satisfactory, prepare the model for deployment (including discussion of limitations and interpretability). If not, revisit feature engineering or model selection.
Note. Feature importance (or SHAP) may be explored on the top-performing models to support interpretability.

EVALUATION METRICS
Since this is a regression task, the evaluation metrics considered are: 

- **Root mean Squared Error (RMSE)**: measures the average magnitude of the errors. It is scale-independent, making it suitable for model comparisons.
- **Mean Absolute Error (MAE)**: captures the mean absolute difference between predictions and actuals values It shares the same unit as the target.
- **R-squared**: shows the proportion of target's variance explained by the model.

The main metric will be RMSE since it *a)* penalized larger errors more heavily; and *b)* is commonly used in models sensitive to significant prediction errors. Therefore, the model having the lowest RMSE will be considered the most-efficient.

Nevertheless, all three metrics will always be assessed together. Examining their relationships can reveal potential issues. More in details:

- **RMSE vs R-squared**: conflicting signals may point out to the need to inspect residuals and re-evaluate the feature set or model complexity.
- **RMSE vs MAE**: RMSE is always >= MAE. The larger difference, the greater the variance in individual erros across the sample.

PERIODIC RE-TRAINING
To ensure top-tier performances, the model is automatically re-trained every 28th of the month. New data is continuously added through an expanding window logic. The re-training procedure is the following:
	- Last month's data are fetched
	- New data are added to the baseline data frame
	- The K-means clustering model is re-trained and new clusters are assigned
	- The updated df is engineered
	- NGboost and Xgboost models are re-trained
	- The models' latest versions are automatically pushed to HF with a training timestamp
This pipeline allows the deployed models to be always up-to-date. 
Moreover, on longer time horizon (6 to 12 months), in-depth data and model drift analysis is conducted. If drift exists, a manual-in-depth re-training and selection is carried out. The model is then updated.
